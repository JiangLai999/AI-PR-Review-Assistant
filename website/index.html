WORKFLOW

从 PR URL 到审查报告的完整路径

这是当前仓库的核心价值，不是大而全的平台，而是一条可解释、可控制、可复用的 CLI 审查流水线。

01

获取 PR

解析 GitHub Pull Request URL，抓取元数据、diff、文件列表和文件内容。

02

过滤文件

跳过纯删除、大文件、二进制和不适合审查的文件，同时支持 force include。

03

构建上下文

把 diff、局部代码窗口、imports、functions、classes 整理成模型上下文。

04

AI 审查

逐文件调用模型输出结构化 findings，并在本地进行成本控制和预算限制。

05

后处理与输出

置信度过滤、去重、渲染成 terminal / markdown / json / GitHub comment。

06

存储与复盘

运行结果写入 SQLite，可继续用 history / stats / chat 工作区追踪。

CAPABILITIES

适合真实团队使用的关键能力

不是堆功能，而是围绕“如何把 AI 审查接进日常开发流程”来组织。

多模型 provider 适配

内置 Anthropics、OpenAI、DeepSeek、Qwen、OpenRouter、Custom 等预设，可切换 provider、model、base URL 与 API format。

配置向导与健康检查

支持交互式配置、配置查看、配置测试、provider health probe、远端模型发现和默认模型切换。

输出与审计留痕

支持结构化 findings、Markdown 报告、JSON 自动化消费以及 SQLite 本地结果持久化。

聊天工作区

在终端内完成 slash command、会话恢复、模型切换和 PR review 触发，降低重复操作成本。

成本与风险控制

在审查执行前后做预算控制、无可审查文件说明、过滤原因输出和更可解释的失败诊断。

CLI 优先的集成方式

更适合本地开发、脚本化调用与 GitHub Actions 等自动化场景，而不是绑定某一个闭源平台。

INSTALL & CONFIG

安装、配置、验证，一次说清楚

这里不只展示安装命令，还把真正影响用户成功率的前置条件、配置策略和常见命令放到同一视图里。

推荐路径

1
安装 CLI

优先使用 pipx install ai-pr-review，隔离依赖且命令直接进 PATH。

2
准备认证信息

至少配置 GITHUB_TOKEN 和一个模型供应商的 API Key。

3
运行配置向导

执行 pr-review config，选择 provider、填写模型和 API 设置。

4
验证配置

执行 pr-review config test 或 pr-review config health --probe。

5
跑一次真实 PR

先用 --only-filter 或 --dry-run 验证，再执行完整审查。
