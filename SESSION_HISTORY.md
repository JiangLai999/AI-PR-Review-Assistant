## 三、关键决策记录

### 决策 1：架构选择

**时间**：00:30
**决策**：真单体架构（FastAPI + BackgroundTasks + SQLite）
**理由**：
- 最小化运维复杂度
- MVP 阶段足够使用
- 后续可扩展

**讨论过程**：
- DeepSeek 质疑了"单体 + Celery"的矛盾
- 指出 GitHub Webhook 的 10 秒限制
- 最终确认使用 BackgroundTasks

---

### 决策 2：模型策略

**时间**：01:00
**决策**：MVP 阶段规则过滤 + 全量 Sonnet
**理由**：
- 避免 Haiku 漏判风险
- 保证代码审查质量
- 成本可控

**讨论过程**：
- 最初考虑 Haiku 分类
- GPT 5.4 指出没有 benchmark 数据
- 最终撤回 Haiku 分类

---

### 决策 3：Prompt 设计

**时间**：01:30
**决策**：双层结构（基础层 400 tokens + 语言层 250 tokens）
**理由**：
- 减少 65% token 消耗
- 平衡成本和效果
- 支持多语言

**讨论过程**：
- 最初 Prompt 约 2000 tokens
- 精简到 650 tokens
- 保留核心检查维度

---

### 决策 4：上下文构建

**时间**：02:00
**决策**：tree-sitter 提取函数/类，三级 fallback
**理由**：
- AST 解析更准确
- 三级 fallback 保证鲁棒性
- 支持多语言

**讨论过程**：
- 最初只做 diff + 前后 10 行
- DeepSeek 指出上下文不足
- 增加 tree-sitter 支持

---

### 决策 5：协作模式

**时间**：02:30
**决策**：Claude Code + GPT 5.4 协作开发
**理由**：
- 多模型协作提高质量
- 不同模型互补
- 透明决策过程

**讨论过程**：
- 最初让 DeepSeek 生成代码
- 用户希望与 GPT 5.4 协作
- 建立新的协作模式

## 五、经验教训

### 成功经验

1. **多方协作** - 不同模型可以互补，提高质量
2. **质疑精神** - 质疑可以帮助发现问题
3. **透明决策** - 决策过程透明，便于追溯
4. **测试驱动** - 测试保证代码质量
5. **渐进开发** - 小步快跑，快速迭代

### 遇到的问题

1. **Windows 控制台问题** - opencode 后台运行可能出错
2. **模型调用超时** - 长时间运行可能超时
3. **方案调整** - 需要根据实际情况调整方案

### 改进建议

1. **优化调用方式** - 避免 Windows 控制台问题
2. **增加超时处理** - 处理长时间运行的任务
3. **灵活调整方案** - 根据实际情况调整
